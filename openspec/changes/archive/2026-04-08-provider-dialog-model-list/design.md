## Context

当前 `ProviderDialog` 只承担供应商基础字段编辑（Name/BaseUrl/ApiKey），不提供模型探测能力。用户需要在编辑阶段直接确认该供应商可用模型，以减少后续测试与切换成本。

约束：
- 手动触发获取（不自动请求）
- 不持久化模型数据
- 必须展示完整返回模型列表并支持搜索
- 失败信息在编辑页内展示

## Goals / Non-Goals

**Goals:**
- 在 `ProviderDialog` 中新增模型获取与列表展示能力
- 按 BaseUrl 规则正确拼接模型接口（`/v1/models` 或 `/models`）
- 对返回结果做本地搜索过滤
- 请求错误可见且可重试

**Non-Goals:**
- 不缓存/落库模型列表
- 不在主窗口显示模型列表
- 不自动定时刷新模型

## Decisions

### 1. 新增独立服务 `ModelDiscoveryService`
- 职责：根据 `Provider` 发送请求、解析模型列表、返回结构化结果
- 原因：避免在 `ProviderDialog.xaml.cs` 混入大量网络逻辑
- 备选：直接写在对话框 code-behind（耦合过高）

### 2. 接口拼接规则固定
- 若 `BaseUrl` 含 `/v1`（路径部分）→ `.../models`
- 否则 → `.../v1/models`
- 原因：兼容两类供应商 BaseUrl 习惯，避免双 `v1`

### 3. UI 交互采用“按钮触发 + 列表过滤”
- “获取模型”按钮进入 loading（禁用 + 文案“获取中...”）
- 下方显示完整模型列表与搜索框；搜索只做本地过滤
- 失败时在列表区上方或下方显示错误消息

### 4. 响应解析策略
- 预期从 JSON `data` 数组读取模型项，优先读取 `id`
- 若结构异常或为空，返回可读错误信息而非静默失败

## Risks / Trade-offs

- [不同供应商返回结构差异] → 先按 `data[].id` 标准结构解析，异常时明确报错
- [模型数量大导致渲染压力] → 首版直接完整展示，后续可加虚拟化优化
- [搜索输入频繁触发过滤] → 仅本地过滤字符串，成本可控

## Migration Plan

- 无数据库迁移
- 发布后仅新增编辑页交互，不影响现有存量数据
