## Context

当前主窗口的“测试”能力仅在点击后通过 MessageBox 反馈，缺少卡片级持续状态。现有 `Provider` 模型和 `Providers` 表没有测试状态字段，无法在重启后保留结果。

约束：
- 状态展示必须极简：只显示状态点，不显示任何文字
- 仅两种可见状态：绿色（可用）和红色（失败）
- 状态需持久化到 SQLite，且兼容已有用户数据库

## Goals / Non-Goals

**Goals:**
- 在供应商名称前显示状态点（绿/红）
- 将最后测试结果持久化到 SQLite（成功/失败）
- 保持现有测试按钮交互和弹窗结果不变
- 对历史数据库无破坏升级

**Non-Goals:**
- 不展示响应时间/错误摘要在卡片上
- 不增加测试历史记录
- 不引入新的测试状态枚举体系（如“测试中”持久化）

## Decisions

### 1. 状态编码使用整型字段 `TestStatus`
- 约定：`0=未知/未测试`、`1=可用(绿)`、`2=失败(红)`
- 原因：实现简单、存储紧凑、与现有 `IsActive` 风格一致
- 备选：字符串状态（可读性高但约束弱）

### 2. 状态点只在 `TestStatus in (1,2)` 时显示
- `1` 显示绿色圆点，`2` 显示红色圆点，`0` 不显示
- 原因：满足“仅红/绿两种状态点”的明确需求
- 备选：默认灰点（会引入第三种视觉状态，不符合需求）

### 3. 持久化路径：复用 `Providers` 表并做增量迁移
- 新增列：`TestStatus INTEGER NOT NULL DEFAULT 0`
- 启动初始化时检测列是否存在，不存在则 `ALTER TABLE` 补列
- 原因：避免新表设计，保持 CRUD 聚合
- 备选：独立测试状态表（结构更复杂，收益低）

### 4. 测试结果写回策略
- 测试成功后写 `TestStatus=1`
- 测试失败后写 `TestStatus=2`
- 编辑供应商配置后重置 `TestStatus=0`
- 原因：避免展示与当前配置不一致的旧状态

## Risks / Trade-offs

- [老库迁移失败] → 启动时先检查列存在再执行 `ALTER TABLE`，失败给出明确错误
- [状态过期误导] → 配置编辑后重置为未知状态
- [仅红/绿信息量有限] → 保留 MessageBox 作为详细反馈通道

## Migration Plan

1. 发布包含新列迁移逻辑的版本
2. 应用启动时自动补齐 `TestStatus` 列（若缺失）
3. 首次迁移后旧记录默认 `TestStatus=0`（不显示状态点）
4. 回滚策略：旧版本忽略新列，不影响基本 CRUD
